# 프로젝트 회고 및 배운 점

이 프로젝트를 진행하면서 React 기반 웹 페이지를 구성하는 방법, 컴포넌트 분리, 라우터 사용, 전역 상태 관리, 커스텀 훅, 폼 관리, 비동기 처리, 폴더 구조 설계 등에 대해 학습할 수 있었습니다.

---

## 1. 컴포넌트 분리

프로젝트 규모가 커지면서 한 페이지 안에 많은 로직과 UI를 모두 담는 것이 점점 어려워졌습니다.
그래서 기능과 역할에 따라 컴포넌트를 최대한 분리하려고 했습니다.

컴포넌트를 분리하니 각 컴포넌트에 맞는 스타일을 적용하기 쉬웠고, 코드의 구조도 더 명확해졌습니다.
하지만 너무 작은 단위까지 컴포넌트를 쪼개다 보니 불필요하게 `props`를 계속 전달해야 하는 상황도 생겼습니다.

이번 경험을 통해 단순히 UI만 보여주는 부분은 한 페이지 안에 두고, 기능에 따라 다른 UI를 보여줘야 하거나 재사용 가능성이 있는 부분은 컴포넌트로 분리하는 것이 더 적절하다는 것을 배웠습니다.

---

## 2. React Router 기능 사용

페이지 이동이 필요한 상황에서는 React Router의 기능을 사용했습니다.

사용자가 첫 페이지에서 버튼을 클릭할 때, 회원가입을 완료했을 때, 로그아웃 버튼을 클릭했을 때, 병원 사이트로 이동해야 할 때에는 `useNavigate`를 사용했습니다.

반면 상단 메뉴바에서 각 페이지로 이동해야 할 때나, 병원 찾기 페이지에서 카테고리를 클릭해 다른 정보를 보여줘야 할 때에는 `Link`를 사용했습니다.

### useNavigate

`useNavigate`는 페이지 이동 전후에 특정 로직을 실행해야 할 때 사용했습니다.
예를 들어 데이터 처리, 조건 검사, 로그인 여부 확인 같은 로직을 실행한 뒤 페이지를 이동시킬 수 있었습니다.

### Link

`Link`는 사용자가 직접 클릭해서 페이지를 이동할 때 사용했습니다.

```jsx
<Link to="/mypage">마이페이지</Link>
```

이번 프로젝트를 통해 `useNavigate`와 `Link`의 차이를 이해할 수 있었습니다.

- `useNavigate`: 함수 안에서 조건에 따라 페이지 이동이 필요할 때 사용
- `Link`: 사용자가 직접 클릭해서 이동하는 메뉴나 버튼에 사용

## 3. 커스텀 훅 사용

웹 페이지를 만들다 보니 팝업 처리, 사용자 정보 표시, 반복되는 상태 관리 로직처럼 여러 컴포넌트에서 반복되는 코드가 많아졌습니다.

처음에는 각 컴포넌트마다 같은 로직을 작성했지만, 코드가 길어지고 유지보수가 어려워졌습니다.
이를 해결하기 위해 반복되는 로직을 커스텀 훅으로 분리했습니다.

커스텀 훅을 사용하니 컴포넌트에서는 필요한 훅을 불러와 사용하기만 하면 되었고, 나중에 로직을 수정해야 할 때도 한 곳만 수정하면 되어서 훨씬 편리했습니다.

또한 컴포넌트 내부 코드가 더 깔끔해져서 가독성도 좋아졌습니다.

---

## 4. Recoil을 활용한 전역 상태 관리

컴포넌트가 많아지면서 `props`로 상태를 계속 전달하는 것이 점점 어려워졌습니다.
불필요한 Props Drilling이 생기면서 코드의 가독성이 떨어지고 유지보수도 어려워졌습니다.

특히 사용자의 정보를 저장해서 마이페이지에서 보여줘야 하거나, AI와 대화한 내용을 리스트로 저장하고 여러 컴포넌트에서 사용해야 하는 상황이 있었습니다.

이때 `useState`만 사용하면 각 컴포넌트마다 상태를 따로 관리해야 해서 복잡했고, 커스텀 훅만으로 관리하기에도 다른 로직과 섞이면 코드가 복잡해지는 문제가 있었습니다.

그래서 Recoil의 `atom`을 사용해 회원 정보와 대화 내용을 저장하는 전역 저장소를 만들었습니다.

이후 여러 컴포넌트에서 필요한 상태를 Recoil Hook으로 가져와 사용할 수 있도록 구성했습니다.

이렇게 구성하니 여러 컴포넌트에서 같은 데이터를 쉽게 공유할 수 있었고, 코드의 가독성과 유지보수성이 좋아졌습니다.

---

## 5. Recoil Atom Effects와 LocalStorage 사용

Recoil의 `atom`은 새로고침을 하면 상태가 초기화되는 단점이 있었습니다.

이로 인해 회원 정보가 사라져서 로그인한 사용자임에도 비회원처럼 처리되는 문제가 있었고, AI와 대화한 기록도 새로고침 후 사라지는 문제가 있었습니다.

이를 해결하기 위해 Recoil의 Atom Effects를 활용해 `localStorage`와 상태를 연결했습니다.

`localStorage`에 기존 값이 있으면 그 값을 불러오고, 값이 변경되면 다시 `localStorage`에 저장되도록 구성했습니다.
또한 로그아웃이나 대화 내역 삭제 시에는 저장된 값을 초기화할 수 있도록 처리했습니다.

이 방식으로 사용자가 새로고침을 해도 회원 정보와 대화 기록이 유지되도록 만들 수 있었습니다.

다만 `localStorage`는 같은 브라우저를 사용하는 다른 사용자가 접근할 수 있고, 중요한 개인정보를 저장하기에는 보안상 위험이 있을 수 있다는 점도 알게 되었습니다.

이 문제를 보완하기 위해 다음 프로젝트에서는 중요한 개인정보는 클라이언트에 저장하지 않고 서버에서 관리하는 방식으로 개선해보고 싶습니다.

또한 인증 관련 정보는 보안 설정이 적용된 쿠키나 세션 스토리지를 사용하는 방법도 함께 고려해볼 수 있습니다.

쿠키를 사용할 경우에는 다음과 같은 설정을 통해 보안을 강화할 수 있습니다.

- `HttpOnly`: JavaScript에서 쿠키에 접근하지 못하도록 설정
- `Secure`: HTTPS 환경에서만 쿠키가 전송되도록 설정

이번 경험을 통해 단순히 데이터를 저장하는 것뿐만 아니라, 보안까지 고려해야 한다는 점을 배울 수 있었습니다.

## 6. useForm Hook 사용

회원가입 페이지를 구현할 때 `useForm` Hook을 사용했습니다.

처음에는 `useState`로 입력값을 하나씩 관리하려고 했지만, 입력 필드가 많아질수록 상태 코드가 길어지고 불필요한 렌더링이 많아지는 문제가 있었습니다.

`useForm`을 사용하니 입력값 관리, 유효성 검사, 에러 메시지 처리 등을 더 직관적으로 구현할 수 있었습니다.

또한 코드가 간결해지고 유지보수하기 쉬워졌으며, 사용자에게 더 명확한 입력 UI를 제공할 수 있었습니다.

---

## 7. 비동기 처리

사용자가 회원가입을 할 때 입력한 정보는 `localStorage`에도 저장되고, `json-server`에도 저장되도록 구현했습니다.
또한 사용자가 AI와 대화할 때에는 AI에게 답변을 요청하고 응답을 받아오는 과정이 필요했습니다.

JavaScript는 한 번에 하나의 작업을 처리하는 싱글 스레드 기반 언어입니다.
따라서 시간이 오래 걸리는 작업을 동기적으로 처리하면 사용자는 화면이 멈춘 것처럼 느낄 수 있습니다.

이를 방지하기 위해 비동기 처리를 사용했습니다.

회원가입 버튼을 클릭하면 서버에 데이터를 저장하면서 다음 동작을 이어갈 수 있도록 했고, AI와 대화할 때도 요청을 보내는 동안 다른 코드가 실행될 수 있도록 구성했습니다.

이를 통해 사용자는 더 빠르고 자연스러운 서비스를 경험할 수 있었습니다.

## 8. 폴더 구조 정리

React 프로젝트를 만들면서 `hooks`, `components`, `pages`처럼 폴더를 나누어 관리했습니다.
하지만 프로젝트 규모가 커질수록 폴더 구조가 복잡해졌고, 코드를 수정하거나 에러를 찾을 때 원하는 파일을 찾기 어려운 문제가 있었습니다.

그러던 중 FSD, 즉 Feature-Sliced Design이라는 폴더 구조를 알게 되었습니다.

FSD는 프로젝트를 기능과 역할에 따라 나누는 구조이며, 일반적으로 다음과 같은 레이어로 구성됩니다.

각 레이어의 역할을 나누면 기능별로 코드를 더 명확하게 관리할 수 있고, 유지보수에도 도움이 될 수 있다고 느꼈습니다.

다음 프로젝트에서는 FSD 폴더 구조를 적용해서 더 직관적이고 유지보수하기 좋은 구조로 프로젝트를 설계해보고 싶습니다.

## 9. Version 2에서 개선하고 싶은 점

현재 병원 찾기 데이터는 임의 데이터로 구성되어 있습니다.
Version 2에서는 실제 공공 데이터를 가져와 병원 정보를 표시하고, 지도 기능까지 연결해보고 싶습니다.

또한 백엔드 서버를 직접 만들어서 CRUD 기능에 대한 이해도를 더 높이고 싶습니다.

추가로 구현해보고 싶은 기능은 다음과 같습니다.

- 공공 데이터 API를 활용한 병원 정보 조회
- 지도 API를 활용한 병원 위치 표시
- 백엔드 서버 구축
- 회원가입 및 로그인 기능 개선
- 사용자 정보 수정 기능
- AI가 사용자 데이터를 불러와 대화에 활용하는 기능
- 마이페이지에서 사용자 정보 수정 기능 추가
- 보안을 고려한 인증 방식 적용

## 마무리

이번 프로젝트를 통해 단순히 화면을 만드는 것뿐만 아니라, 프로젝트 구조를 어떻게 설계해야 하는지, 상태를 어떻게 관리해야 하는지, 반복되는 로직을 어떻게 분리해야 하는지에 대해 배울 수 있었습니다.

또한 데이터를 저장하고 관리할 때는 사용자 경험뿐만 아니라 보안도 함께 고려해야 한다는 점을 알게 되었습니다.

다음 프로젝트에서는 이번에 배운 내용을 바탕으로 더 안정적이고 유지보수하기 쉬운 구조를 적용해보고 싶습니다.
